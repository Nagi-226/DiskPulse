# DiskPulse v0.6.0 — Cross-Platform Performance Foundation

> **Status**: Implementation in progress | **Base**: v0.5.0 | **Target**: v0.6.0

## Theme

v0.5.0 made every feature work together. v0.6.0 makes it **fast and everywhere** — replacing the polling watcher with native kernel events, adding hard-link-aware dedup and sparse-file detection, and shipping first-class Linux + macOS support through a unified platform trait architecture.

**Motto**: "One codebase, three platforms, native performance."

---

## Architecture: 6-Platform-Trait Matrix

All OS-specific code is isolated behind 6 traits. Each trait has exactly one implementation per platform. Adding a new platform or swapping a performance backend means writing one new `impl` block — zero changes to business logic.

| # | Trait | Status | Purpose | Windows | Linux | macOS |
|---|-------|--------|---------|---------|-------|-------|
| 1 | `DiskInfoProvider` | Existing, extended | Drive space, list drives, FS type | `GetDiskFreeSpaceExW` | `statvfs` | `statfs` |
| 2 | `FsWatcher` | **New** | Real-time FS event monitoring | `ReadDirectoryChangesW` | `inotify` | `FSEvents` |
| 3 | `DirScanner` | Existing `ScanStage`, upgraded | Directory walk & measurement | `JwalkStage` (default), `MftStage` (reserve) | `LinuxWalkStage` | `MacOsWalkStage` |
| 4 | `CleanupProvider` | **New** | Safe file deletion (Recycle Bin / Trash) | `SHFileOperationW` | `trash-rs` | `NSFileManager` |
| 5 | `FileMetaAnalyzer` | **New** | Hard links, sparse files, file identity | `GetFileInformationByHandle` | `statx` | `stat` |
| 6 | `SystemInfo` | **New** | OS metadata, CPU, RAM, data dirs | `GetSystemInfo` | `uname + /proc` | `sysctl` |

### Trait Layout Strategy

Each trait follows the same pattern already established by `ScanStage` and `DiskInfoProvider`:

- Trait definition lives in `src-tauri/src/platform/mod.rs` (co-located, not scattered across crates)
- Each platform impl gets its own file: `platform/windows.rs`, `platform/linux.rs`, `platform/macos.rs`
- `#[cfg(target_os)]` compile-time dispatch — zero runtime overhead
- Technical reserves (e.g. `MftStage`) live in `platform/windows_mft.rs` — compiled but not wired until activated

```
src-tauri/src/platform/
├── mod.rs              ← 6 trait definitions
├── windows.rs          ← Windows impls (JwalkStage, ReadDirChanges, SHFileOp, GetFileInfo)
├── windows_mft.rs      ← MFT direct scan (技术储备 — compiled, gated behind feature flag)
├── linux.rs            ← Linux impls (inotify, statvfs, trash-rs)
├── macos.rs            ← macOS impls (FSEvents, statfs, NSFileManager)
└── common.rs           ← Shared types: WatcherGuard, FileIdentity, TrashResult
```

### ScanStrategy Selector

```rust
pub enum ScanStrategy {
    Auto,                  // Best available for current platform
    Jwalk,                 // Cross-platform jwalk (current default)
    #[cfg(target_os = "windows")]
    Mft { admin: bool },   // NTFS MFT direct read (技术储备)
}

impl ScanStrategy {
    pub fn resolve(&self) -> Box<dyn ScanStage> {
        match self {
            #[cfg(target_os = "windows")] ScanStrategy::Auto => Box::new(JwalkStage),
            #[cfg(target_os = "linux")]   ScanStrategy::Auto => Box::new(LinuxWalkStage),
            #[cfg(target_os = "macos")]   ScanStrategy::Auto => Box::new(MacOsWalkStage),
            // MftStage only wired when feature-flag + admin detected
            ...
        }
    }
}
```

### Phase 2: Windows Native Performance (v0.5.4 — v0.5.6)

> **Goal**: Swap Windows backend from polling/generic to native kernel APIs.
> Zero frontend changes — just drop-in backend replacements.

#### v0.5.4 - ReadDirectoryChangesW Native Watcher (In Progress)

Replace `WindowsPollWatcher` with `WindowsNativeWatcher`:

- `CreateFileW` on each watched directory with `FILE_FLAG_OVERLAPPED | FILE_FLAG_BACKUP_SEMANTICS`
- `ReadDirectoryChangesW` with `FILE_NOTIFY_CHANGE_FILE_NAME | SIZE | LAST_WRITE`
- Overlapped I/O + completion routine via `BindIoCompletionCallback` or dedicated thread with `GetQueuedCompletionStatus`
- Batch events with debounce (same `FsChangeBatch` format as polling — frontend unchanged)
- `cancel` via `CancelIo` + `CloseHandle`

New `windows` crate features needed: `Win32_System_IO` (for `OVERLAPPED`, `CreateFileW`, `ReadDirectoryChangesW`)

Tech spec:
| Metric | Polling (current) | ReadDirectoryChangesW (v0.5.4) |
|--------|-------------------|-------------------------------|
| Event latency | 2s (poll interval) | < 50ms (kernel push) |
| CPU (idle) | ~1% (sleep + snapshot) | ~0% (blocked on kernel) |
| CPU (active) | 5-10% (snapshot diff) | <1% (kernel filters) |
| Missed events | Possible (between polls) | Zero (kernel buffer) |

**Watcher fallback chain**: `ReadDirectoryChangesW` → (if unavailable) `WindowsPollWatcher` → (if unavailable) error

Verify: `cargo test` + manual smoke: create/delete files in monitored dir, verify events appear < 1s

#### v0.5.5 - Hard Link Aware Duplicate Detection (Complete)

- Add `FileMetaAnalyzer::file_identity()` call before SHA-256 hashing in duplicates pipeline
- If two files have same `FileIdentity { volume_serial, file_index }` → same file (hard link) → skip
- Report hard link savings: "X hard links found, Y bytes already shared on disk"
- `FileMetaAnalyzer::hard_link_count()` → surface in `LargeFileFinder` UI as "Links: N"

Duplicates pipeline change:
```
Before:  Size group → First 4KB hash → Full SHA-256
After:   Size group → FileIdentity check → (skip same-identity) → First 4KB hash → SHA-256
```

Performance impact: `GetFileInformationByHandle` is a single kernel call (~μs), cheaper than hashing 4KB.

Verify: `cargo test duplicates`, add 1 test: files with same FileIdentity are skipped

#### v0.5.6 - Sparse File Detection (In Progress)

- `FileMetaAnalyzer::is_sparse()` → check `FILE_ATTRIBUTE_SPARSE_FILE` via `GetFileInformationByHandle`

- Surface in `LargeFileFinder`: "12.0 GB apparent / 3.2 GB on disk (sparse)"
- Surface in `AgingAnalysis`: sparse files flagged separately — they're not "wasted" space

No cleanup changes needed — Recycle Bin handles sparse files correctly.
