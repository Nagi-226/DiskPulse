## Phase 1: 基础打磨（v0.6.1 ~ v0.6.2）

> 目标：关闭 v0.3.x 遗留的技术债务。v0.6.1 的流式扫描是整个 v0.7.0 的数据管道基础。

### v0.6.1 — 流式增量扫描（关闭 v0.3.3 缺口）

**问题现状**：当前 `scan_drive_dirs` 使用 rayon 并行收集，全部完成后才一次性返回。前端必须等待全量扫描完成才能看到任何结果。

**改造目标**：

| 指标 | 当前 (v0.6.0) | 目标 (v0.6.1) |
|------|--------------|--------------|
| 首个结果可见时间 | 全量扫描完成后（5-10s） | <500ms（首个目录完成即推送） |
| 内存峰值 | ~80MB | <50MB（流式 + 增量释放） |
| 取消响应 | <500ms | <200ms（每批检查取消标志） |
| 增量更新 | 不支持 | Watcher 事件触发单目录重扫 |

**技术方案**：

1. **ScanStage trait 升级** — 新增 `execute_streaming()` 方法，返回 `mpsc::Receiver<ScanBatch>`：
   ```rust
   struct ScanBatch { dirs: Vec<DirInfo>, batch_index: u32, is_complete: bool }
   trait ScanStage {
       fn execute_streaming(&self, ctx: &ScanContext) -> Receiver<ScanBatch>;
   }
   ```
2. **JwalkStage 实现** — 每完成一个顶层目录的 `measure_directory()`，立即发送一批
3. **IPC 事件** — 新增 `scan-batch` 事件（ScanProgress 的 Phase 枚举增加 `Streaming` 变体）
4. **前端增量渲染** — Treemap 在 `Streaming` 阶段逐批更新，不等待 `Complete`
5. **增量重扫** — Watcher 检测到变更目录 → `scan_directory(path)` → 局部刷新 Treemap

**涉及文件**：
- 修改: `src-tauri/src/scanner/mod.rs`, `src-tauri/src/platform/mod.rs`, `src-tauri/src/platform/windows.rs`
- 修改: `src-tauri/src/lib.rs`, `src/types.ts`
- 修改: `src/App.tsx`, `src/hooks/useDriveScan.ts`
- 新增测试: 流式推送测试、增量重扫测试、取消响应时间测试

**验证**：`cargo test` (90+ tests), `npm run typecheck`, 手动：扫描 500GB 驱动，观察 Treemap 逐批出现

## Phase 2: 深度性能（v0.6.3 ~ v0.6.4）

> 目标：激活技术储备 + 后台常驻运行。从"可用"到"飞起来"。

### v0.6.3 — MFT 主文件表直读扫描

**技术背景**：当前使用 walkdir/jwalk 逐目录遍历 — 每个目录和文件都需要单独 FS 系统调用。NTFS 的 `$MFT` 是一张包含所有文件元数据的数据库表，直接读取可一次 I/O 获取百万级文件信息。

v0.6.0 中 `MftStage` 已编译但被 feature-gated（`mft-scanner` feature flag），等待激活条件成熟。

**激活方案**：

| 条件 | v0.6.3 判断 |
|------|------------|
| ntfs-rs 依赖 | **绕过** — 直接用 windows crate 的 FSCTL 调用 |
| 管理员权限 | **新增** — 检测进程权限，MFT 需要管理员权限 |
| 性能基准 | **新增** — 同一驱动上对比 MFT vs Jwalk |
| 降级策略 | **保持** — MFT 失败自动降级到 JwalkStage |

**技术方案**：

```rust
// 1. 打开卷句柄
CreateFileW(r"\\.\C:", ...)
// 2. FSCTL_GET_NTFS_VOLUME_DATA → NTFS_VOLUME_DATA_BUFFER  
// 3. FSCTL_ENUM_USN_DATA → 批量 USN_RECORD（路径/大小/时间戳）
// 4. 解析为 DirInfo 并流式推送（复用 v0.6.1 的 ScanBatch 通道）

// 策略选择器升级
ScanStrategy::Auto {
    // 1. 检测管理员权限
    // 2. 有权限 → MftStage
    // 3. 无权限 → JwalkStage (fallback)
    // 4. MFT 失败 → JwalkStage (降级)
}
```

**性能预期**：

| 场景 | JwalkStage | MftStage (预期) |
|------|-----------|----------------|
| 500GB HDD, 500K 文件 | 5-8s | 1-2s |
| 1TB SSD, 1M 文件 | 3-5s | <1s |
| 2TB HDD, 2M 文件 | 15-20s | 2-4s |

**风险与回退**：
- ⚠️ USN 记录是文件变更历史，不是完整文件列表。已删除文件可能还在，新文件可能未进入。
- 🛡️ MftStage 定位为**快速近似扫描**（标注 "approximate"），JwalkStage 为**精确扫描**（默认）。用户可选择：速度优先 vs 精度优先。
- Feature flag `mft-scanner` 默认关闭。

**涉及文件**：
- 修改: `src-tauri/src/platform/windows_mft.rs`, `src-tauri/src/platform/mod.rs`
- 修改: `src-tauri/src/scanner/mod.rs`, `src-tauri/Cargo.toml`
- 新增测试: 权限检测测试、MFT 正确性测试、降级回退测试
- 新增: 性能基准对比 (MFT vs Jwalk)

**验证**：`cargo test`, 手动：管理员运行，对比 MFT 和 Jwalk 扫描时间

## Phase 3: 智能层（v0.6.5 ~ v0.6.6）

> 目标：从"看到数据"到"理解数据"。纯 Rust 统计模型驱动，零外部 ML 依赖。

### v0.6.5 — 智能异常检测

**设计理念**：当前预测使用简单线性回归 — 假设磁盘使用匀速增长。实际上磁盘使用是非线性的、有季节性的。v0.6.5 引入两个纯 Rust 统计模型。

**模型 1 — Holt-Winters 季节性预测**：

替代当前 OLS 线性回归。三指数平滑同时追踪水平、趋势、季节性三个分量：

```rust
level[t]    = α * (y[t] / seasonal[t-L]) + (1-α) * (level[t-1] + trend[t-1])
trend[t]    = β * (level[t] - level[t-1]) + (1-β) * trend[t-1]
seasonal[t] = γ * (y[t] / level[t]) + (1-γ) * seasonal[t-L]
预测: forecast[t+h] = (level[t] + h*trend[t]) * seasonal[t-L+h]
默认周期 L=7（天），自动检测更好的周期
```

| 指标 | 当前 (OLS) | v0.6.5 (HW) |
|------|----------|------------|
| 趋势 | ✅ 线性 | ✅ 指数平滑（自适应） |
| 季节性 | ❌ 不支持 | ✅ 日/周/月周期 |
| 置信区间 | 固定 95% | ✅ 动态误差估计 |
| 最小数据点 | 3 | 2L（14 个快照） |

**模型 2 — Modified Z-Score 异常检测器**：

```rust
MAD = median(|xi - median(X)|)  // 中位数绝对偏差
Mi = 0.6745 * (xi - median(X)) / MAD
异常阈值：|Mi| > 3.5 → 标记为异常

// 检测维度：
// 1. 空间消耗速率（GB/天 突然飙升）
// 2. 文件数量增长率（大量新文件被创建）  
// 3. 特定目录突变（某个目录突然膨胀）
```

**异常类型**：

| 类型 | 描述 | 建议动作 |
|------|------|---------|
| 🟡 速率异常 | 消耗速率 > 正常值 3.5σ | 检查大型下载/日志膨胀 |
| 🔴 突发异常 | 单次快照变化 > 正常值 10σ | 立即告警 — 可能是恶意软件/误操作 |
| 🟠 目录热点 | 某目录增长速率显著高于历史 | 引导用户关注该目录 |
| 🔵 模式偏离 | 季节性模式突然改变 | 可能是系统配置变更 |

**新 IPC**：`detect_anomalies(drive: String) -> AnomalyReport`
**新组件**：`AnomalyCard.tsx` — 仪表盘异常摘要卡片
**升级**：`predict_disk_usage` 返回格式扩展，增加 Holt-Winters 的季节分量和动态置信区间

**涉及文件**：

### v0.6.6 — 智能推荐引擎 v2

**从 v1 到 v2**：v0.3.6 的推荐使用静态加权评分。v2 升级为**动态上下文感知**评分 — 同样的临时文件在磁盘紧张时优先级更高。

**三个新维度**：

| 维度 | 数据来源 | 作用方式 |
|------|---------|---------|
| **紧迫性 Urgency** | HW 预测 + 异常检测 | 磁盘越接近满 → 推荐权重越高（动态倍率 1.0x ~ 3.0x） |
| **模式学习 Pattern** | cleanup_logs 表 | 用户频繁清理的路径类别 → 排名提升 |
| **关联发现 Correlation** | 跨模块数据关联 | 同一目录被多个检测器标记 → 提升为"高价值清理目标" |

**评分公式 v2**：
```rust
score = (w_risk * S_risk + w_age * S_age + w_size * S_size
         + w_dup * S_dup + w_safety * S_safety)
         * urgency_multiplier     // 1.0 ~ 3.0x
         * pattern_boost           // 1.0 ~ 1.5x
         + correlation_bonus       // 跨模块关联加分

// urgency_multiplier
days_until_full = prediction.days_until_full
if days_until_full < 7:  3.0
if days_until_full < 30: 2.0
if days_until_full < 90: 1.5
else: 1.0
```

**磁盘健康 v2** — 从单一分数拆为 4 维度雷达图：

| 子评分 | 含义 | 输入 |
|--------|------|------|
| 空间容量 Space | 剩余空间 vs 总容量 | DiskInfoProvider |
| 垃圾堆积 Waste | 可清理空间 vs 已用空间 | Risk Report + Duplicates |
| 增长趋势 Trend | 消耗速率 + 异常信号 | HW 预测 + 异常检测 |
| 碎片老化 Age | 僵尸文件占比 + 超大文件 | Aging + Large Files |

**涉及文件**：
- 修改: `src-tauri/src/recommendations/mod.rs`, `src-tauri/src/db/mod.rs`
- 修改: `src/components/RecommendationCard.tsx`, `src/components/DiskHealthGauge.tsx`
- 修改: `src/types.ts`
- 新增测试: 紧迫性倍率、行为模式学习、健康维度分拆

### v0.6.7 — 多设备 Dashboard

**使用场景**：开发者在主力机上看到所有设备的磁盘状态、接收任何设备的告警。

**架构设计**：

```
┌──────────────────────────────────────────────────────┐
│                   Hub Device (主力机)                  │
│  ┌────────────────────────────────────────────────┐  │
│  │  WebSocket Server (localhost:19740)             │  │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐        │  │
│  │  │ 本机引擎 │  │ 设备注册 │  │ 消息路由 │        │  │
│  │  └─────────┘  └────┬────┘  └────┬────┘        │  │
│  └────────────────────┼────────────┼──────────────┘  │
│                       │            │                  │
└───────────────────────┼────────────┼──────────────────┘
                        │ WebSocket  │
                  ┌─────┴──────┐
                  │ LAN (局域网) │
                  └──┬───────┬──┘
            ┌────────┴──┐ ┌──┴──────────┐
            │  笔记本    │ │  NAS        │
            │  Linux     │ │  Linux      │
            │  WS Client │ │  WS Client  │
            └───────────┘ └─────────────┘
```

**核心设计决策**：

| 决策项 | 方案 | 理由 |
|--------|------|------|
| 通信协议 | WebSocket + JSON | Web 标准，Tauri 原生支持，零额外依赖 |
| 消息格式 | 复用 Tauri IPC 命令格式 | 同一套 invoke 协议，WS 只是传输层 |
| 服务发现 | mDNS (Bonjour) | 零配置局域网发现，用户无需输入 IP |
| 安全 | 配对令牌 + 局域网限定 | 首次连接需在 Hub 上确认（6 位配对码） |
| 远程命令白名单 | 只读命令全开，写入命令需确认 | 远程清理必须 Hub 确认 |

**WS 消息信封**（在现有 IPC 命令外套一层路由信息）：
```json
// Client → Hub
{ "id": "uuid", "type": "invoke", "target": "device-abc",
  "command": "scan_drive_meta", "args": { "drive": "C:" } }

// Hub → Client（响应）
{ "id": "uuid", "type": "response", "source": "device-abc",
  "ok": true, "data": { ... } }

// Hub → Client（事件推送）
{ "id": "uuid", "type": "event", "source": "device-abc",
```

**安全模型**：

| 安全层级 | 机制 |
|---------|------|
| 网络边界 | WS Server 绑定 localhost，远程通过 Hub 中转 |

| 命令白名单 | 只读命令（scan/list/get）自动放行；写入命令需 Hub 确认 |
| 远程清理 | Hub 弹窗确认 → 仅 LOW 风险 → 仅 Recycle Bin |

## 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| MFT USN 数据不完整 | 中 | 中 | 标注 "approximate"，Jwalk 始终是 fallback |
| Holt-Winters 需要 ~2 周快照数据 | 低 | 中 | 降级逻辑：数据不足时回退到 OLS |
| Windows Service 权限问题 | 中 | 中 | 详细日志 + 错误提示；Service 不执行清理 |
| WebSocket Hub 跨平台兼容 | 中 | 中 | 先在 Windows 验证，再测 Linux/macOS |
| mDNS 在企业网络中被禁用 | 低 | 低 | 支持手动输入 IP:Port 作为 fallback |
| v0.7.0 范围过大 | 中 | 高 | MVP-first，每个版本独立可交付。可在任何 Phase 边界裁剪 |

## 设计原则

1. **稳定优先** — 每个版本独立可交付，绝不破坏已有功能
2. **纯 Rust 智能** — 统计模型零外部 ML 依赖，编译快、体积小
3. **安全红线不变** — 所有清理走 Recycle Bin，Service 模式不执行清理
4. **渐进智能** — 从填坑 → 提速 → 统计模型 → 用户行为学习，层层递进
5. **跨平台一致** — 所有新功能通过 6-trait 平台抽象层，一次编写三平台运行
