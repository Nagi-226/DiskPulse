## Roadmap (4 Sprints)

### Sprint 1: Disk Space Alerts ‚úÖ

| Feature | Status |
|---------|--------|
| Low space notification via tauri-plugin-notification (percentage or absolute GB) | ‚úÖ |
| Sudden growth detection with configurable percent and time window | ‚úÖ |
| Background monitor thread with AtomicBool cancel token | ‚úÖ |
| Alerts settings tab (enable, threshold, growth params) | ‚úÖ |
| Dashboard in-app alert toast banner | ‚úÖ |
| 6 new AppSettings alert fields + 4 unit tests | ‚úÖ |

### Sprint 2: Large File Finder ‚úÖ

- **File-level scan**: backend complete via `find_large_files_with_progress_and_cancel(drive, min_size, limit)` ‚Äî walks drive, collects individual files above threshold, uses `BinaryHeap<Reverse<FileEntry>>` to track top-N
- **Progress + cancel**: backend complete ‚Äî emits `large-file-progress` events during scan and supports `cancel_large_file_scan`
- **UI**: frontend complete ‚Äî "Large Files" tab with sortable table (path, size, age, cleanup status) and "Add to Cleanup" integration
- **Integration**: frontend complete ‚Äî selected large files are added to cleanup preview through `additionalItems` and reuse the `preview_cleanup` pipeline

### Sprint 3: Disk Usage Prediction ‚úÖ

- **Linear regression** on SQLite snapshot history (no new dependency ‚Äî simple OLS implementation) ‚úÖ
- **Forecast**: "at current growth rate, disk will reach 95% full in ~X days" with confidence score ‚úÖ
- **Trend chart**: dashed forecast line extending existing ECharts trend chart in History page ‚úÖ
- **Prediction card**: rendered on dashboard between drive ring and treemap ‚úÖ

### Sprint 4: Scheduled Auto-Cleanup ‚úÖ

- **Scheduler thread**: backend complete ‚Äî daily/weekly/monthly at configured time; scan ‚Üí classify ‚Üí clean LOW-risk items ‚Üí save report ‚Üí notify
- **Safety invariant**: backend complete ‚Äî only LOW risk levels, never MEDIUM/HIGH. All deletes still via Recycle Bin.
- **Auto-cleanup history**: complete - new `auto_cleanup_reports` SQLite table plus History page report timeline
- **Settings UI**: complete - Automation tab with enable toggle, frequency, time, min-free-GB-to-skip, LOW-only invariant, Save Automation, and Run Now
- **Dashboard**: complete - `AutoCleanupStatus` card showing schedule state, next run, last result, and manual Run Now

## Dependency & Sequencing

Recommended order: **Sprint 1 ‚Üí Sprint 3 ‚Üí Sprint 2 ‚Üí Sprint 4**

- S1 establishes notification + background thread patterns that S4 reuses
- S3 is quick (low effort, high visibility) and uses only existing DB snapshots
- S2 is the most work-intensive (new scanner mode with file-level tracking)
- S4 is the capstone that ties alerts + cleanup + scheduling together

Zero new crate dependencies. All features use already-included crates.
