## Roadmap (4 Sprints)

### Sprint 1: Disk Space Alerts ✅

| Feature | Status |
|---------|--------|
| Low space notification via tauri-plugin-notification (percentage or absolute GB) | ✅ |
| Sudden growth detection with configurable percent and time window | ✅ |
| Background monitor thread with AtomicBool cancel token | ✅ |
| Alerts settings tab (enable, threshold, growth params) | ✅ |
| Dashboard in-app alert toast banner | ✅ |
| 6 new AppSettings alert fields + 4 unit tests | ✅ |

### Sprint 2: Large File Finder 🔄

- **File-level scan**: new `find_large_files(drive, min_size, limit)` function — walks drive, collects individual files above threshold, uses `BinaryHeap<Reverse<FileEntry>>` to track top-N
- **Progress + cancel**: emits `large-file-progress` events during scan, supports cancellation
- **UI**: new "Large Files" tab in dashboard — sortable table (path, size, age, risk) with "Add to Cleanup" integration
- **Integration**: selected large files can be added to cleanup preview (reuses `preview_cleanup` pipeline)

### Sprint 3: Disk Usage Prediction ✅

- **Linear regression** on SQLite snapshot history (no new dependency — simple OLS implementation) ✅
- **Forecast**: "at current growth rate, disk will reach 95% full in ~X days" with confidence score ✅
- **Trend chart**: dashed forecast line extending existing ECharts trend chart in History page ✅
- **Prediction card**: rendered on dashboard between drive ring and treemap ✅

### Sprint 4: Scheduled Auto-Cleanup 🔄

- **Scheduler thread**: daily/weekly/monthly at configured time — scan → classify → clean LOW-risk items → save report → notify
- **Safety invariant**: only LOW (and optionally MEDIUM) risk levels, never HIGH. All deletes still via Recycle Bin.
- **Auto-cleanup history**: new `auto_cleanup_reports` SQLite table, displayed in History page
- **Settings UI**: new Automation tab — frequency, time, risk level checkboxes, min-free-GB-to-skip, "Run Now" button
- **Dashboard**: `AutoCleanupStatus` card showing next run / last result

## Dependency & Sequencing

Recommended order: **Sprint 1 → Sprint 3 → Sprint 2 → Sprint 4**

- S1 establishes notification + background thread patterns that S4 reuses
- S3 is quick (low effort, high visibility) and uses only existing DB snapshots
- S2 is the most work-intensive (new scanner mode with file-level tracking)
- S4 is the capstone that ties alerts + cleanup + scheduling together

Zero new crate dependencies. All features use already-included crates.
