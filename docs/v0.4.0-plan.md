# DiskPulse v0.4.0 — Extensible Intelligence Platform

> **Status**: Planning | **Base**: v0.3.0 | **Target**: TBD

## Theme

Transform DiskPulse from a "monitoring & cleanup tool" into an **extensible disk intelligence platform** — with plugin-style architecture, multi-dimensional space analysis, and guided optimization.

## Design Philosophy

Every core system is refactored to a **trait + registry** pattern so that future features can be added by implementing a trait and registering it — without touching core code.

## Extensibility Architecture

### Six Extension Points

| # | Extension Point | Current Problem | v0.4.0 Design | Landing Version |
|---|----------------|----------------|---------------|-----------------|
| 1 | **Risk Rule Registry** | 16 rules hardcoded in `risk/mod.rs` | `trait RiskRule` + registry pattern; new rule = implement trait + register | v0.3.7 |
| 2 | **Scanner Plugin Pipeline** | Scan logic monolithic in `scanner/mod.rs` | `trait ScanStage`; custom walk/filter/measure phases | v0.3.3 |
| 3 | **Notification Channel** | Windows native notification only | `trait NotifyChannel`; pluggable: WinNotify / Webhook / Silent Log | v0.3.8 |
| 4 | **Cleanup Provider** | Hard dependency on `SHFileOperationW` | `trait CleanupProvider`; Windows→RecycleBin, Linux→trash-rs, macOS→Trash | v0.3.9 |
| 5 | **i18n Resource Bundle** | No i18n | JSON resource bundle + `I18nProvider`; new language = new JSON file | v0.3.1 |
| 6 | **Theme Token System** | CSS hardcoded | CSS custom properties + theme variable map; new theme = new values | v0.3.2 |

### Key Design Decisions

1. **Trait over hardcode** — All extension points use Rust trait interfaces
2. **React Context + Provider** — ThemeProvider, I18nProvider wrap component tree
3. **New feature = new file (or crate)** — Never modify core modules for new features
4. **Config-driven** — Risk rules, notification channels, cleanup strategies read from config/SQLite
5. **Versioned data schemas** — Settings schema carries version number; new fields have defaults

### v0.3.3 — Scan Performance Overhaul

**Goal**: Refactor scan pipeline to pluggable architecture; implement jwalk parallel traversal and streaming results.

| Layer | Details |
|-------|---------|
| **jwalk Integration** | Replace `walkdir` with `jwalk`; multi-threaded parallel deep directory walking; add `jwalk` to `Cargo.toml` |
| **Streaming Results** | Use `std::sync::mpsc::channel` instead of `Vec::collect()`; frontend renders as data streams in |
| **Incremental Scan** | FS watcher detects change → re-scan only affected top-level dir → merge update into existing results |
| **ScanStage Trait** | Define `trait ScanStage { fn execute(&self, ctx: &ScanContext) -> ScanOutput; }` — Walk / Filter / Measure three phases replaceable |
| **Memory Optimization** | Eliminate intermediate collections; full iterator chain; target peak memory < 100MB (500GB drive) |
| **Cancel Granularity** | Check cancel token between each `ScanStage` execution; cancel response < 500ms |

**Performance Targets**:

| Metric | v0.3.0 Current | v0.3.3 Target |
|--------|---------------|---------------|
| Full drive scan (500GB) | 5-15s | 2-5s |
| Large file find (500GB) | 76s | 15-25s |
| Peak memory | ~200MB | < 100MB |
| Incremental update (post FS change) | Full rescan | < 1s |
| Cancel response | 1-3s | < 500ms |

**Extensibility**: New scan type (duplicate detection, aging analysis) = implement new `ScanStage` + register in pipeline.
`ScanContext` carries shared state (cancel token, progress sender, cache handle), extensible by field.

**Verification**: `cargo test` + `cargo bench` (new benchmarks) + `cargo clippy -- -D warnings`

## Phase 2: Intelligence (v0.3.4 — v0.3.6)

### v0.3.4 — Duplicate File Detection

**Goal**: Find identical-content files at different paths; quantify recoverable wasted space.

| Layer | Details |
|-------|---------|
| **New Module** | `src-tauri/src/duplicates/mod.rs` |
| **Three-Phase Detection** | ① Byte-size grouping (skip unique-size files) → ② First 4KB hash pre-filter (fast exclusion) → ③ SHA-256 full-file hash confirmation (exact match) |
| **Configuration** | `min_file_size` (default 1MB, skip small files), `max_depth`, `exclude_patterns` |
| **Progress Events** | `duplicate-scan-progress` IPC event: current phase, files processed, duplicate groups found |
| **Cancel** | `cancel_duplicate_scan` command + `DUPLICATE_SCAN_CANCEL` static AtomicBool |
| **Return Type** | `Vec<DuplicateGroup> { group_id, total_size_wasted, files: Vec<FileEntry> }` |
| **IPC Commands** | `scan_duplicates(drive, min_size)` → `Vec<DuplicateGroup>`, `cancel_duplicate_scan()` |
| **Frontend** | `DuplicateFinder.tsx` — new Dashboard tab; summary card + duplicate groups table sorted by wasted space |
| **Smart Selection** | Auto-select "keep newest, delete rest" per group (user can manually adjust) |
| **Cleanup Integration** | "Add to Cleanup" button, reuses `CleanupPreview` pipeline |

**Extensibility**: Hash algorithm configurable via `HashAlgorithm` enum (SHA-256 default, switchable to xxHash or BLAKE3).
`DuplicateDetector` trait: `fn detect(config) -> Vec<DuplicateGroup>`, swappable implementation.

**Verification**: 3+ unit tests (size grouping, hash confirmation, mid-scan cancel); `npm run typecheck` + `npm run build:web`

### v0.3.5 — File Aging Analysis

**Goal**: Map file access/modification time distributions; find "zombie files" (long unaccessed) and "growth hotspots" (recently active directories).

| Layer | Details |
|-------|---------|
| **New Module** | `src-tauri/src/aging/mod.rs` |
| **Time Dimensions** | Creation time, last access time, last modification time (Windows FILE_BASIC_INFO) |
| **Aging Buckets** | <7d / 7-30d / 1-3mo / 3-6mo / 6-12mo / 1-3yr / >3yr |
| **Two Analysis Modes** | ① Zombie Finder: `last_access > 180d` files aggregated by directory; ② Growth Hotspots: `last_modified < 30d` files aggregated by directory growth rate |
| **Return Type** | `AgingReport { buckets: Vec<AgeBucket>, zombies_total_size, hotspots: Vec<Hotspot> }` |
| **IPC Commands** | `analyze_file_aging(drive)` + `cancel_aging_scan()` |
| **Frontend** | `AgingAnalysis.tsx` — stacked bar chart (ECharts) showing space distribution by aging bucket |
| **Interaction** | Click bucket → drill-down to file list; zombie files highlighted with estimated recoverable space |
| **Cleanup Integration** | Zombie file list supports "Add to Cleanup" |

**Extensibility**: `TimeClassifier` trait: default fixed-day bucketing, extensible to dynamic bucketing (data-distribution based).
`AgingMetric` enum: LastAccess / LastModified / CreationTime, new dimensions addable.

**Verification**: 2+ unit tests (bucketing logic, zombie detection); `npm run typecheck` + `npm run build:web`

### v0.3.6 — Smart Recommendation Engine + Disk Health Score

**Goal**: Aggregate all detection capabilities from previous versions; give users a "one-click health check → Top cleanup recommendations" experience.

| Layer | Details |
|-------|---------|
| **New Module** | `src-tauri/src/recommendations/mod.rs` |
| **Scoring Model** | Composite score 0-100: `score = w1*risk_factor + w2*age_factor + w3*duplicate_factor + w4*size_factor + w5*safety_factor` |
| **Data Sources** | Risk engine (16 rules), duplicates detector (groups), aging analyzer (buckets), scanner (directory sizes) |
| **Return Type** | `Vec<Recommendation> { rank, item, score, reason, estimated_size, action: CleanAction }` |
| **Disk Health Score** | Composite index 0-100: `health = f(free_pct, growth_rate, duplicate_waste, zombie_pct, alert_status)` |
| **IPC Commands** | `get_recommendations(drive)` → `Vec<Recommendation>`, `get_disk_health(drive)` → `DiskHealth` |
| **Frontend** | `RecommendationCard.tsx` — Dashboard card Top 5 recommendations + "Run Full Health Check" button |
| **Health Gauge** | Semi-circular gauge (ECharts gauge), red→yellow→green color scale, click to expand detailed breakdown |

**Extensibility**: Scoring weights configurable (Settings → Recommendations Tab), tunable by user or future versions.
`ScoringFunction` trait: default weighted linear, extensible to non-linear model.
Pluggable recommendation sources: `RecommendationSource` trait — new detection modules implement this to feed into recommendation engine.

**Verification**: 3+ unit tests (score calculation, ranking, empty-data edge cases); `npm run typecheck` + `npm run build:web` + manual UI verification

## Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| jwalk compilation complexity on Windows | Medium | Benchmark vs walkdir first; keep walkdir as fallback feature flag |
| SHA-256 hashing makes duplicate detection slow on large files | Medium | Use first-4KB pre-filter; expose hash algorithm choice; consider xxHash fast path |
| Cross-platform abstraction touches every OS-specific call | Low | Scope to trait definition only; Linux/macOS implementations deferred to v0.5.0 |
| i18n retrofitting touches every UI string | Medium | Do it first (v0.3.1) to minimize retrofitting cost |
| Too many new modules increases build time | Low | Each module is independently compiled; cargo incremental builds handle this |
| Custom risk rules may have unintended matches | Medium | Real-time path testing in rule editor; safety pipeline still blocks system paths |

- v0.3.8: Wizard + notifications — UX capstone, reuses all prior modules
- v0.3.9: CLI + platform — prepares for cross-platform without committing to full implementations

Zero new external crate dependencies except `jwalk` (v0.3.3) and `react-i18next` (v0.3.1).
